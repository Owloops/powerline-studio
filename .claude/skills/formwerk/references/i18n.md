# Internationalization (i18n)

Formwerk treats internationalization as a first-class concern. Beyond translating labels and messages, it handles locale-specific issues:

- `input[type="number"]` only accepts Latin numerals -- problematic for non-Latin locales/keyboard layouts
- Some keyboard layouts lack semantically correct commas and decimal separators
- Arrow key shortcuts often don't respect locale directionality
- HTML5 validation messages appear in the browser's language, not the website's

## Not an i18n Library

Formwerk does **not** provide translation. Use a dedicated i18n library like `vue-i18n` alongside Formwerk for label/message translation.

## Locale Configuration

### Setting Locale

```ts
import { configure } from '@formwerk/core'

configure({
	locale: 'en-US',
})
```

Default locale: `en-US`

### Locale Auto-Detection

Formwerk auto-detects the locale from the `<html lang="...">` attribute at import time. It does **not** use the browser's locale (which may differ from the website's target audience).

- No need to disable -- manual `configure()` always takes precedence
- **Not available in SSR** -- set locale manually via `configure()` in SSR environments

## Directionality

Formwerk determines text directionality using the `Intl.Locale` API based on the configured locale. This sets the default `dir` property for composables.

### Disable Auto-Detection

```ts
configure({
	detectDirection: false,
})
```

- You can override `dir` per-composable via the `dir` prop
- Components using arrow keys (e.g., sliders) adapt functionality to direction
- Styling for RTL/LTR is your responsibility

## Usage with Vue i18n

Set locale where you initialize `vue-i18n`. Formwerk accepts a **reactive** locale value and auto-updates directionality on change.

```ts
import { createI18n } from 'vue-i18n'
import { configure } from '@formwerk/core'

const i18n = createI18n({
	locale: 'en-US',
})

configure({
	// Reactive if legacy: false in vue-i18n options
	locale: i18n.global.locale,
})
```

## Usage with Nuxt i18n

No `<html>` tag in SSR -- must set locale manually. Use a Nuxt plugin to ensure configuration happens before Formwerk is used.

### Using the Locale Ref

Pass a reactive locale value for automatic directionality updates:

```ts
import { configure } from '@formwerk/core'

export default defineNuxtPlugin((nuxtApp) => {
	configure({
		locale: nuxtApp.$i18n.locale,
	})
})
```

### Using Nuxt Hooks

Listen to `i18n:localeSwitched` to update locale when `setLocale` is called:

```ts
import { configure } from '@formwerk/core'

export default defineNuxtPlugin((nuxtApp) => {
	nuxtApp.hook('i18n:localeSwitched', ({ newLocale }) => {
		configure({
			locale: newLocale,
		})
	})
})
```

## HTML Validation Language Mismatch

HTML5 validation messages are generated in the browser's default language, not the website's. No JS API exists to influence the message text.

**Recommended workaround:** Disable HTML5 validation message generation and use Standard Schema validation for localized messages.

- You can still pass `required` and other HTML validation attributes
- They will **not** affect validity or messages once HTML5 validation is disabled
